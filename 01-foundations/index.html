CORE CONCEPT

What is Reactive Programming?

A paradigm built around asynchronous data streams and the propagation of change.

Definition: Reactive programming is about building systems that react to events as they happen — instead of asking for data, you subscribe to streams and let the data come to you.
Everything is a Stream
  • Mouse clicks & keyboard input
  • HTTP responses & WebSocket messages
  • Timer ticks & intervals
  • State changes in your application
  • Database query results
The Reactive Manifesto
  • Responsive fast and consistent
  • Resilient stays alive on failure
  • Elastic scales under load
  • Message-Driven async boundaries

WHY REACTIVE

Why Reactive? The Problem

Callbacks and Promises work — but they break down at scale and complexity.

😩 Callback Hell
getUser(id, (err, user) => {
  if (err) handleError(err);
  getOrders(user.id, (err, orders) => {
    if (err) handleError(err);
    getProduct(orders[0].id, (err, product) => {
      if (err) handleError(err);
      // deeply nested, untestable,
      // hard to read & maintain 😫
    });
  });
});
😐 Promises Help, But...
getUser(id)
  .then(user => getOrders(user.id))
  .then(orders => getProduct(orders[0].id))
  .then(product => display(product))
  .catch(handleError);
// Better! But still limited...
// ❌ Only one value
// ❌ Starts immediately (eager)
// ❌ Cannot be cancelled
// ❌ No backpressure control
Promise Limitations at a Glance
Scenario Promise Need
User types in a search box ❌ Single value Stream of keystrokes over time
User navigates away from page ❌ Cannot cancel Cancel in-flight HTTP request
Stock price ticker ❌ One-shot Continuous stream of updates
WebSocket messages ❌ Not lazy Subscribe / unsubscribe on demand
Compose 3 async operations ⚠️ Awkward chaining Powerful composable operators
Observables fill the gap: lazy, cancellable, composable streams of zero-to-many values — all with built-in error handling and completion signals.

DESIGN PATTERN — GoF

Observer Pattern

One subject maintains a list of observers and notifies them automatically when state changes.

Structure
Subject
subscribe() · unsubscribe() · notify()
Observer A Observer B Observer C
  • Subject — holds the list of observers
  • Observer — reacts to notifications
  • subscribe() — opt in to updates
  • unsubscribe() — opt out (prevent leaks!)
  • Loose coupling — subject doesn't know observers
Real-world examples
  • DOM addEventListener / removeEventListener
  • React useEffect with event listeners
  • Node.js EventEmitter
  • RxJS Observable.subscribe()
  • Vue watch / watchEffect
This is the backbone of Reactive Programming. Observables are the Observer Pattern taken to its logical extreme — a standardized, composable, cancellable push stream.
Implementation — JavaScript
JavaScript 
class Subject {
  constructor() {
    this.observers = new Set();
  }

  subscribe(observer) {
    this.observers.add(observer);
    return () => this.observers.delete(observer); // returns unsubscribe fn
  }

  notify(value) {
    this.observers.forEach(obs => obs(value));
  }
}

// Usage
const subject = new Subject();

const unsubA = subject.subscribe(v => console.log('Observer A:', v));
const unsubB = subject.subscribe(v => console.log('Observer B:', v));

subject.notify('hello');   // Observer A: hello  |  Observer B: hello
unsubA();                  // A unsubscribes
subject.notify('world');   // Observer B: world  (A no longer notified)

Iterator Pattern

Provides a way to access elements of a collection sequentially without exposing its internals.

How It Works
Consumer
──next()──▶
Iterator
──▶
{ value, done }
  • Consumer controls when to pull next value
  • next() returns { value, done }
  • Hides the collection's internal structure
  • Often lazy — only computes when asked
  • [Symbol.iterator] makes any object iterable in JS
Observer vs Iterator
ObserverIterator
DirectionPushPull
Who controls timingProducerConsumer
Lazy by default❌✅
Async support✅❌

Observable = Observer + Iterator merged into one push-based lazy stream.

Implementation — JavaScript
JavaScript 
// Custom iterable — pull-based, lazy range
const range = {
  [Symbol.iterator]() {
    let current = 1, end = 3;
    return {
      next() {
        return current <= end
          ? { value: current++, done: false }
          : { value: undefined, done: true };
      }
    };
  }
};

for (const n of range) {
  console.log(n); // 1, 2, 3
}

// Generator syntax — the modern shortcut
function* numbers(start, end) {
  for (let i = start; i <= end; i++) {
    yield i; // pauses here until consumer calls next()
  }
}

console.log([...numbers(1, 5)]); // [1, 2, 3, 4, 5]

CORE CONCEPT

Pull vs Push

The fundamental question: who decides when data arrives?

The 2 × 2 Model
Single value Multiple values
Pull — Consumer controls Function
const x = getValue()		 Iterator
for (const n of range) …
Push — Producer controls Promise
fetch('/api').then(…)		 Observable
fromEvent(btn, 'click').pipe(…)
Pull — Consumer Decides
JavaScript 
// Pull: I ask, I get
const value = getTemperature(); // sync function

// Pull: I iterate when I want
for (const item of database.findAll()) {
  process(item); // consumer controls pace
}
  • Synchronous by nature
  • Consumer drives the flow
  • Easy to reason about
Push — Producer Decides
JavaScript 
// Push: it notifies me when ready
fetch('/api/data').then(data => {
  handle(data); // arrives when it's ready
});

// Push: events arrive on their own schedule
fromEvent(window, 'resize').subscribe(e => {
  updateLayout(e); // producer decides timing
});
  • Inherently asynchronous
  • Producer drives the flow
  • Natural for events & I/O
Key insight: Reactive programming is about embracing push. Instead of polling or asking repeatedly, you subscribe once and let data flow to you — enabling more efficient, responsive systems.

COMPARISON

Promises vs Observables

Why Observables are a strict superset — and when each one wins.

Feature Comparison
FeaturePromiseObservable
Values emittedSingle✅ Zero to many (stream)
Execution❌ Eager (starts immediately)✅ Lazy (starts on subscribe)
Cancellable❌ No✅ Yes — unsubscribe()
Synchronous possible❌ Always async✅ Can be sync or async
Error handling.catch(fn)catchError() + retry(n)
Composition.then() chainingpipe() + 100+ operators
Retry logicManual (recursive)retry(3) / retryWhen()
MulticastAll share one resultSubjects, share(), publish()
Backpressure❌ None✅ throttle / debounce / buffer
Promise — when it fits well
JavaScript 
// ✅ Great for: one-shot async operations
const user = await fetch('/api/user')
  .then(r => r.json());

// Simple, readable, widely understood
const [users, posts] = await Promise.all([
  fetchUsers(),
  fetchPosts()
]);
Observable — where it shines
JavaScript 
// ✅ Great for: streams, events, composition
const search$ = fromEvent(input, 'input').pipe(
  map(e => e.target.value),
  debounceTime(300),     // wait for pause in typing
  distinctUntilChanged(), // skip repeated values
  switchMap(q => searchApi(q)), // cancel stale calls
  retry(2),              // retry on error
  catchError(() => of([]))  // fallback on failure
);

const sub = search$.subscribe(setResults);
// cleanup: sub.unsubscribe();
Rule of thumb: Use a Promise for "get me one thing once." Use an Observable for "give me things as they happen — and let me control the flow."
