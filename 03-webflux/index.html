MOTIVATION

Why WebFlux?

Traditional servlet stacks struggle with high concurrency. WebFlux solves this with a non-blocking event-loop model.

The C10K Problem — Serving 10,000+ Concurrent Connections
😴 Spring MVC (Blocking)
Request A → Thread-1 → waiting for DB ···········→ response
Request B → Thread-2 → waiting for DB ···········→ response
Request C → Thread-3 → waiting for DB ···········→ response
Request N → ❌ no threads left → queue → timeout
  • One thread tied up per in-flight request
  • Threads are expensive (~1MB stack each)
  • Thread pool exhausted under load → 503s
⚡ Spring WebFlux (Non-Blocking)
Request A → event loop → register DB call → free thread
Request B → event loop → register DB call → free thread
Request C → event loop → register DB call → free thread
DB ready → resume pipeline → send response
  • Few threads handle thousands of requests
  • Thread is freed while waiting for I/O
  • Scales to 10x+ concurrent connections
When to Use WebFlux
  • I/O-bound workloads (DB, HTTP, queues)
  • High concurrency requirements
  • Streaming data (SSE, WebSocket)
  • Microservices calling many other services
When NOT to Use WebFlux
  • CPU-bound work (reactive won't help)
  • Team will call blocking libraries anyway
  • Simple CRUD with low traffic
  • Codebase already in Spring MVC
Key rule: Never block inside a reactive pipeline. Calling Thread.sleep(), blocking JDBC, or any blocking I/O inside a map() will freeze the event loop and eliminate all performance benefits.

Mono & Flux

The two core types of Project Reactor — the reactive library powering Spring WebFlux.

REST API

REST API — ProductController

The actual controller from this project — showing how Mono and Flux map naturally to REST semantics.

ProductController.java — from this project
Java · Spring WebFlux 
@RestController
@RequestMapping("/products")
public class ProductController {

    private final Map<String, Product> products = new ConcurrentHashMap<>(Map.of(
        "p-100", new Product("p-100", "Keyboard", BigDecimal.valueOf(120), "hardware"),
        "p-200", new Product("p-200", "Monitor",  BigDecimal.valueOf(899), "hardware"),
        "p-300", new Product("p-300", "Reactive Spring", BigDecimal.valueOf(59), "books")
    ));

    // Mono<T> = 0 or 1 result  →  maps to: 200 OK or 404 Not Found
    @GetMapping("/{id}")
    public Mono<Product> getById(@PathVariable String id) {
        return Mono.justOrEmpty(products.get(id));
    }

    // Flux<T> = 0..N results  →  returns a JSON array
    @GetMapping
    public Flux<Product> getAll() {
        return Flux.fromIterable(products.values());
    }

    // Server-Sent Events — stream items to the client in real time
    @GetMapping(value = "/stream",
                produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<Product> streamProducts() {
        return Flux.fromIterable(products.values())
            .delayElements(Duration.ofMillis(500)); // emit one every 500ms
    }

    // Reactive request body — no blocking deserialization
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public Mono<Product> create(@RequestBody Mono<Product> body) {
        return body.map(product -> {
            products.put(product.id(), product);
            return product;
        });
    }
}

OPERATORS

Operators & Error Handling

Project Reactor ships with the same operator model as RxJS — fluent, composable, testable.

Common Operators
Java 
// map — synchronous transform (1:1)
Flux.range(1, 5)
    .map(n -> n * n);   // 1, 4, 9, 16, 25

// filter — keep matching elements
Flux.range(1, 6)
    .filter(n -> n % 2 == 0);   // 2, 4, 6

// flatMap — async 1:N (parallel)
userIds.flatMap(id -> fetchUser(id));

// concatMap — async 1:N (sequential, ordered)
userIds.concatMap(id -> fetchUser(id));

// zip — combine latest from two sources
Mono.zip(namesMono, emailsMono,
    (name, email) -> name + " <" + email + ">");
Error Handling
Java 
// onErrorReturn — fallback value
fetchProduct(id)
    .onErrorReturn(Product.UNKNOWN);

// onErrorResume — fallback Mono/Flux
fetchProduct(id)
    .onErrorResume(err -> cache.get(id));

// retry with backoff
fetchProduct(id)
    .retryWhen(Retry.backoff(3, Duration.ofSeconds(1)));

// doOnError — side-effect (log), then rethrow
fetchProduct(id)
    .doOnError(err -> log.error("Failed", err))
    .onErrorReturn(Product.UNKNOWN);
Real Pipeline — Combining Operators
Java · Spring WebFlux 
// A realistic service method: fetch products by category, enrich with stock info
public Flux<ProductDetail> getProductsWithStock(String category) {
    return productRepository.findByCategory(category)          // Flux<Product>
        .filter(p -> p.price().compareTo(BigDecimal.ZERO) > 0) // skip free items
        .flatMap(product ->                                     // parallel stock fetch
            stockService.getStock(product.id())
                .map(stock -> new ProductDetail(product, stock))
                .onErrorResume(err -> {
                    log.warn("Stock unavailable for {}", product.id());
                    return Mono.just(new ProductDetail(product, Stock.UNKNOWN));
                })
        )
        .take(50)                                              // limit results
        .cache(Duration.ofSeconds(30));                        // cache for 30s
}
